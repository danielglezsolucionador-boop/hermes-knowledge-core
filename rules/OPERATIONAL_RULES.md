# OPERATIONAL_RULES

## PURPOSE

Defines mandatory operational rules for the ecosystem.

These rules override improvisation.

---

# CORE_RULES

1. Stability before expansion.
2. No direct production experimentation.
3. Every change requires validation.
4. Every deployment must be reversible.
5. Runtime integrity is priority.
6. No uncontrolled architecture growth.
7. No undocumented operational systems.
8. Human approval required for critical operations.
9. Failures must remain observable.
10. Recovery capability is mandatory.

---

# DEVELOPMENT_RULES

1. Small isolated changes only.
2. No giant refactors.
3. No architecture rewrites during stabilization.
4. One operational objective per task.
5. Validate locally before deployment.
6. Production validation required after deploy.
7. No “magic fixes”.
8. Every operational change requires rollback path.

---

# DEPLOYMENT_RULES

Before every deploy:

- backup
- clean commit
- deployment validation
- runtime validation
- rollback capability
- production verification

After every deploy:

- validate endpoints
- validate dashboard
- validate runtime
- validate Telegram
- validate database
- validate logs

---

# RUNTIME_PROTECTION

Protected systems:

- runtime loop
- recovery system
- PostgreSQL integrity
- operational monitoring
- task execution system

Changes to protected systems require:
- explicit approval
- isolated validation
- rollback plan

---

# FRONTEND_RULES

Frontend must:
- never break runtime
- remain operationally clear
- avoid fake interactions
- avoid fake metrics
- avoid dead buttons

Operational functionality overrides visual effects.

---

# AI_RULES

AI systems must:
- remain observable
- avoid hallucinated operational states
- avoid fake confirmations
- expose failures clearly
- prioritize operational truth

---

# PHASE_RULES

No phase advances unless:

- operational stability exists
- recovery is validated
- failures remain controlled
- rollback remains possible

---

# KNOWLEDGE_CORE_RULES

Knowledge Core is operational truth source.

Rules:
- markdown only
- structured format
- no speculative architecture
- no marketing language
- no duplicated definitions

---

# FAILURE_PROTOCOL

If instability appears:

1. stop deployments
2. isolate issue
3. validate logs
4. restore stable state
5. validate recovery
6. document incident
7. redeploy only after validation

---

# LONG_TERM_RULE

The ecosystem must evolve through controlled operational growth.

Uncontrolled complexity is forbidden.
