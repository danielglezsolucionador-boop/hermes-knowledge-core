# SUBFASE 4.2.2 — TASK CLAIMING

## OBJETIVO REAL

Construir el sistema que permite a Hermes reclamar ownership operacional de una task válida.

Esta subfase NO ejecuta tasks todavía.

SOLO permite:

- reclamar tasks
- bloquear ownership
- evitar doble pickup
- preparar ejecución futura

━━━━━━━━━━━━━━━━━━━
REGLA CRÍTICA
━━━━━━━━━━━━━━━━━━━

Hermes NO puede:

- reclamar la misma task dos veces
- permitir doble ejecución
- reclamar tasks inválidas
- reclamar tasks corruptas
- perder control ownership
- generar race conditions

El claiming debe ser:

- atómico
- estable
- observable
- controlado

━━━━━━━━━━━━━━━━━━━
OBJETIVO TÉCNICO
━━━━━━━━━━━━━━━━━━━

Construir la capa de ownership operacional del runtime Hermes.

Esta subfase será usada después por:

- execution runtime
- retries
- timeout control
- orchestration
- monitoring
- recovery systems

━━━━━━━━━━━━━━━━━━━
IMPLEMENTAR
━━━━━━━━━━━━━━━━━━━

# 1. Task Claiming Layer

Crear sistema que permite reclamar tasks disponibles.

Responsabilidades:

- seleccionar candidate válida
- reclamar ownership
- bloquear task
- preparar transición operacional

SIN ejecutar todavía.

━━━━━━━━━━━━━━━━━━━

# 2. Allowed Claim States

Hermes SOLO puede reclamar tasks:

- pending

PROHIBIDO reclamar:

- doing
- done
- failed
- cancelled
- corrupted

━━━━━━━━━━━━━━━━━━━

# 3. Atomic Claiming

Implementar claiming atómico.

Debe garantizar:

- una task = un runner
- no doble ownership
- no race conditions
- consistencia operacional

PROHIBIDO:

- claiming parcial
- ownership ambiguo
- updates inseguros

━━━━━━━━━━━━━━━━━━━

# 4. Runtime Ownership

Registrar ownership operacional:

- runner_id
- runtime_id
- claimed_at
- claim_state

Ownership REAL solamente.

━━━━━━━━━━━━━━━━━━━

# 5. Claim Validation

Validar antes de reclamar:

- task sigue pending
- task sigue válida
- task no está lockeada
- runtime autorizado
- task no cambió estado

━━━━━━━━━━━━━━━━━━━

# 6. Claim State Transition

Implementar transición controlada:

pending
↓
claimed

SIN saltos inválidos.

PROHIBIDO:

- pending → doing
- pending → done
- pending → failed

━━━━━━━━━━━━━━━━━━━

# 7. Claim Limits

Implementar límites estrictos:

- max concurrent claims
- max claim attempts
- max claim rate
- max stale claims

Evitar:

- saturación runtime
- ownership masivo
- loops agresivos

━━━━━━━━━━━━━━━━━━━

# 8. Claim Logging

Loggear:

- task reclamada
- runner owner
- timestamp claim
- conflictos ownership
- claims rechazados
- errores claiming

SIN spam logs.

━━━━━━━━━━━━━━━━━━━

# 9. Failed Claim Handling

Preparar manejo seguro cuando:

- claim falla
- ownership conflict existe
- task desaparece
- task cambia estado

El runtime debe:

- mantenerse estable
- evitar crash
- liberar recursos
- continuar operativo
