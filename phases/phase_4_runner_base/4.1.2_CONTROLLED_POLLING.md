# SUBFASE 4.1.2 — CONTROLLED POLLING

## OBJETIVO REAL

Construir el sistema controlado de polling operacional para Hermes runtime.

Esta subfase crea:
la capa que permite detectar trabajo pendiente SIN generar comportamiento agresivo o inestable.

IMPORTANTE:

Todavía NO se ejecutan tasks.

Todavía NO existe integración IA.

Todavía NO existe orchestration avanzada.

El objetivo es:
permitir que Hermes consulte estado operacional de forma controlada,
estable
y observable.

━━━━━━━━━━━━━━━━━━━
REGLA CRÍTICA
━━━━━━━━━━━━━━━━━━━

El polling NO puede convertirse en:

* busy loop
* spam runtime
* consumo excesivo
* ejecución descontrolada

Hermes debe controlar:
- frecuencia polling
- volumen operacional
- estabilidad runtime
- observabilidad del polling

━━━━━━━━━━━━━━━━━━━
OBJETIVO TÉCNICO
━━━━━━━━━━━━━━━━━━━

Construir el sistema base de polling reutilizable para:

* task pickup
* runtime monitoring
* observability
* orchestration futura
* execution coordination

SIN mezclar ejecución todavía.

━━━━━━━━━━━━━━━━━━━
IMPLEMENTAR
━━━━━━━━━━━━━━━━━━━

# 1. Controlled Polling Cycle

Implementar ciclo polling controlado.

Responsabilidades:

* consultar estado operacional
* buscar tasks pendientes
* mantener frecuencia estable
* evitar polling agresivo

SIN ejecutar tasks todavía.

━━━━━━━━━━━━━━━━━━━

# 2. Polling Interval Control

Implementar intervalos configurables:

* polling_sleep
* polling_interval
* controlled wait

Objetivo:
mantener estabilidad operacional.

PROHIBIDO:
loops sin pausa.

━━━━━━━━━━━━━━━━━━━

# 3. Pending Task Detection

Implementar detección básica de:

* pending tasks
* runtime state
* queue visibility

SIN modificar tasks todavía.

━━━━━━━━━━━━━━━━━━━

# 4. Polling Visibility

Exponer:

* last_poll_time
* polling_iteration
* tasks_detected
* polling_status

Datos REALES solamente.

━━━━━━━━━━━━━━━━━━━

# 5. Empty Queue Handling

Si no existen tasks:

* mantener loop estable
* evitar errores innecesarios
* continuar polling controlado

PROHIBIDO:
crash por queue vacía.

━━━━━━━━━━━━━━━━━━━

# 6. Polling Logging

Loggear solamente:

* polling started
* polling completed
* tasks detectadas
* polling errors críticos

SIN spam logs.

━━━━━━━━━━━━━━━━━━━

# 7. Polling Error Containment

Si ocurre error durante polling:

* preservar runtime
* evitar crash total
* mantener observabilidad
* continuar loop seguro si es posible

PROHIBIDO:
stop silencioso del polling.

━━━━━━━━━━━━━━━━━━━

# 8. Runtime Stability Protection

El polling debe respetar:

* runtime health
* CPU control
* observabilidad
* timing estable

SIN degradar FastAPI ni PostgreSQL.

━━━━━━━━━━━━━━━━━━━
VALIDACIONES OBLIGATORIAS
━━━━━━━━━━━━━━━━━━━

Debe validarse:

✅ polling inicia
✅ polling detecta tasks
✅ polling con queue vacía
✅ polling mantiene timing estable
✅ polling NO genera busy loop
✅ observabilidad polling visible
✅ logs mínimos correctos
✅ runtime sigue healthy
✅ FastAPI estable
✅ PostgreSQL estable

━━━━━━━━━━━━━━━━━━━
PROHIBIDO
━━━━━━━━━━━━━━━━━━━

NO implementar:

* task execution
* retries
* AI providers
* orchestration
* task locking
* task claiming
* autonomous execution
* multiprocessing complejo
* self-healing
* retries automáticos

detectar trabajo pendiente,
mantener polling controlado,
preservar estabilidad runtime
y mantener observabilidad operacional.

SIN ejecutar tasks todavía.
