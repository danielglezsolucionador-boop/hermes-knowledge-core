# SUBFASE 4.2.1 — TASK DISCOVERY

## OBJETIVO REAL

Construir el sistema que permite a Hermes descubrir tasks disponibles para ejecución.

Esta subfase NO ejecuta tasks todavía.

SOLO descubre:
qué tasks podrían ejecutarse.

━━━━━━━━━━━━━━━━━━━
REGLA CRÍTICA
━━━━━━━━━━━━━━━━━━━

Hermes NO puede:

- recoger tasks incorrectas
- descubrir tasks duplicadas
- ejecutar tasks inválidas
- procesar tasks corruptas
- ignorar estado operacional

El discovery debe ser:
controlado,
determinístico,
observable.

━━━━━━━━━━━━━━━━━━━
OBJETIVO TÉCNICO
━━━━━━━━━━━━━━━━━━━

Construir la capa de descubrimiento operacional de tasks.

Esta subfase será usada después por:

- claiming
- execution
- retry systems
- timeout systems
- orchestration runtime
- observability

━━━━━━━━━━━━━━━━━━━
IMPLEMENTAR
━━━━━━━━━━━━━━━━━━━

# 1. Task Discovery Layer

Crear sistema que detecta tasks disponibles.

Responsabilidades:

- buscar tasks ejecutables
- validar estado task
- filtrar tasks inválidas
- preparar candidates para pickup

━━━━━━━━━━━━━━━━━━━

# 2. Allowed Task States

Hermes SOLO puede descubrir tasks:

- pending

NO descubrir:

- doing
- done
- failed
- cancelled
- corrupted

━━━━━━━━━━━━━━━━━━━

# 3. Controlled Querying

Implementar querying controlado.

IMPORTANTE:

NO traer:
- toda la base completa
- payloads gigantes
- tasks históricas innecesarias

Debe existir:
- límites
- ordering
- control operacional

━━━━━━━━━━━━━━━━━━━

# 4. Task Ordering

Priorizar discovery por:

1. prioridad operacional
2. fecha creación
3. retry state
4. execution eligibility

NO usar orden aleatorio.

━━━━━━━━━━━━━━━━━━━

# 5. Discovery Validation

Validar:

- task tiene status válido
- task tiene payload válido
- task no está corrupta
- task puede ejecutarse
- task pertenece al runtime actual

━━━━━━━━━━━━━━━━━━━

# 6. Discovery Limits

Implementar límites estrictos:

- max tasks discovery
- max batch size
- max payload inspection
- max query duration

Evitar:
- saturación runtime
- memory spikes
- polling agresivo

━━━━━━━━━━━━━━━━━━━

# 7. Discovery Logging

Loggear:

- tasks descubiertas
- cantidad encontrada
- duración query
- filtros aplicados
- tasks ignoradas
- errores discovery

SIN spam excesivo.

━━━━━━━━━━━━━━━━━━━

# 8. Empty Queue Handling

Preparar manejo seguro cuando:

NO existan tasks pendientes.

El runtime debe:

- mantenerse estable
- esperar correctamente
- NO entrar en busy loop

━━━━━━━━━━━━━━━━━━━
