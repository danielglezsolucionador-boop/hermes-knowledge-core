# 5.1.4_EXECUTION_CONTEXT_MEMORY

## PURPOSE

Define cómo Hermes preserva, recupera y reutiliza contexto operacional durante execution runtime.

Este sistema permite:

- mantener continuidad
- recordar estado execution
- recuperar checkpoints
- preservar trazabilidad
- mantener coherencia operacional
- evitar pérdida de contexto

Es la base de la memoria operacional de Hermes.

---

# OBJECTIVES

Hermes debe poder:

- preservar contexto execution
- recuperar contexto previo
- mantener continuidad runtime
- reutilizar información relevante
- recordar checkpoints
- preservar trazabilidad operacional

---

# EXECUTION CONTEXT CONCEPT

El execution context representa:

- el estado vivo de trabajo
- la memoria operacional activa
- la continuidad runtime
- la información necesaria para continuar execution

Hermes NO debe:

- perder contexto crítico
- olvidar checkpoints
- mezclar contexts diferentes
- continuar con contexto corrupto

---

---

# CONTEXT FLOW

## 1. CONTEXT LOAD

Hermes carga:

- contexto previo
- checkpoints
- logs
- estado execution
- archivos relacionados

---

## 2. CONTEXT PRESERVATION

Durante execution:

- actualiza memoria runtime
- registra cambios
- preserva continuidad
- guarda checkpoints

---

## 3. CONTEXT RECOVERY

Si execution falla:

- recuperar último checkpoint
- restaurar estado
- restaurar lifecycle
- continuar desde punto seguro

---

## 4. CONTEXT CLOSE

Al terminar:

- guardar contexto final
- preservar trazabilidad
- registrar completion
- preparar contexto futuro

---

# REQUIRED MEMORY RULES

## CHECKPOINT SAFETY

Hermes debe registrar checkpoints seguros.

Especialmente:

- antes auditoría
- antes retry
- antes cambios críticos
- antes continuation

---

## CONTEXT ISOLATION

Hermes NO puede:

- mezclar tasks
- compartir contextos corruptos
- reutilizar contextos inválidos
- sobrescribir execution activa

---

## RECOVERY SAFETY

Recovery debe:

- preservar trazabilidad
- restaurar lifecycle
- restaurar estado válido
- evitar corrupción runtime

---
