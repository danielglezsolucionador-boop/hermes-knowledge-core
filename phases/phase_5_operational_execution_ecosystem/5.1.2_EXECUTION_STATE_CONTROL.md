# 5.1.2_EXECUTION_STATE_CONTROL

## PURPOSE

Define cómo Hermes controla y preserva estados operacionales durante execution runtime.

Este sistema permite:

- controlar lifecycle execution
- preservar continuidad operacional
- detectar bloqueos
- detectar errores
- manejar transitions runtime
- evitar execution corruption

Es la base del comportamiento operacional estable de Hermes.

---

# OBJECTIVES

Hermes debe poder:

- controlar execution states
- cambiar estados correctamente
- preservar consistencia runtime
- detectar estados inválidos
- bloquear transitions peligrosas
- mantener continuidad operacional
- permitir recuperación segura

---

# EXECUTION STATE CONCEPT

Un execution state representa:

- la situación operacional actual
- el estado runtime activo
- el punto real del lifecycle execution

Cada task debe tener:

- un único estado principal
- transitions válidas
- trazabilidad completa
- consistencia operacional

---

# REQUIRED STATES

## CREATED

Task creada.
Todavía no ejecutada.

---

## INITIALIZING

Hermes:

- carga contexto
- prepara runtime
- valida dependencias
- inicia execution session

---

## RUNNING

Execution activa.

Hermes:

- trabaja task
- ejecuta acciones
- monitorea runtime
- registra progreso

---

## WAITING_APPROVAL

Esperando aprobación humana.

Hermes:

- NO continúa automáticamente
- preserva estado actual
- mantiene contexto vivo

---

## WAITING_AUDIT

Esperando auditoría.

Hermes:

- congela continuation
- preserva execution context
- espera validación externa

---

## RETRY_PENDING

Retry autorizado pendiente.

---

## BLOCKED

Existe bloqueo operacional.

Ejemplos:

- dependencia rota
- conflicto arquitectura
- provider unavailable
- validación fallida

---

## FAILED

Execution falló.

Debe existir:

- error reason
- logs
- contexto preservado
- recovery visibility

---

## COMPLETED

Task terminada correctamente.

---

# STATE TRANSITIONS

## VALID TRANSITIONS

CREATED
→ INITIALIZING

INITIALIZING
→ RUNNING

RUNNING
→ WAITING_AUDIT

RUNNING
→ WAITING_APPROVAL

RUNNING
→ BLOCKED

RUNNING
→ FAILED

WAITING_AUDIT
→ RUNNING

WAITING_APPROVAL
→ RUNNING

RETRY_PENDING
→ RUNNING

RUNNING
→ COMPLETED

---

# INVALID TRANSITIONS

PROHIBIDO:

FAILED
→ COMPLETED

BLOCKED
→ COMPLETED

COMPLETED
→ RUNNING

FAILED
→ RUNNING
sin recovery autorizado

---

# STATE SAFETY RULES

Hermes NO debe:

- saltar estados
- ejecutar transitions inválidas
- sobrescribir estado activo
- perder execution context
- continuar después de FAILED sin autorización
- autoaprobar transitions críticas

---

# BLOCK DETECTION

Hermes debe detectar:

- dependency conflicts
- invalid architecture
- runtime corruption
- provider failure
- audit rejection
- approval rejection
- retry overflow

---

# FAILURE VISIBILITY

Cada FAILED debe registrar:

- error reason
- execution step
- archivos afectados
- último checkpoint
- provider usado
- timestamp
- audit status

---

# HUMAN VISIBILITY

El humano debe poder ver:

- estado actual
- transición actual
- execution progress
- runtime issues
- bloqueos
- auditorías pendientes

---
